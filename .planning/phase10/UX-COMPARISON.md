## Tool 1: gsd:progress

Source file read: ~/.claude/get-shit-done/workflows/progress.md (full)

### What it does well

**Pattern 1 — Structured status header with named sections**
Every progress report opens with a fixed-field block:

    # [Project Name]
    Progress: [bar]
    Profile: [quality/balanced/budget/inherit]
    Discuss mode: [value]

Then named sections in consistent order: Recent Work / Current Position /
Key Decisions Made / Blockers / Pending Todos / What's Next.
Users always know where to look for each class of information.

**Pattern 2 — Inline routing with a single recommended next action**
Every route (A through F) ends with exactly one primary command on its own
line, followed by a subdued `/clear first` hint, then an "Also available:"
secondary list. The primary action is the recommendation; alternatives are
never buried. Example:

    /gsd:execute-phase {phase} ${GSD_WS}
    /clear first -- fresh context window

    Also available:
    - /gsd:plan-phase {phase} -- skip execution, plan directly

**Pattern 3 — Verification debt surfaced as a warning table, not a blocker**
When outstanding UAT debt exists, gsd:progress inserts a "Verification Debt"
table between "What's Next" and the route suggestion. It lists phase, file,
and pending/skipped counts. It calls out the debt visibly but explicitly marks
it a WARNING and lets routing continue. The user sees the cost before choosing.

**Pattern 4 — Progress bar from tooling, not prose**
The progress bar is generated by a CLI tool (`gsd-tools.cjs progress bar`)
and injected as a single token. The orchestrator never assembles it by hand.
This keeps the rendered format deterministic and testable.

| # | Takeaway | Disposition |
|---|----------|-------------|
| G1 | Fixed-field status header (named sections, consistent order) | Addressed by 10A-wf.3 and 10A-st.3 -- "Where am I" block and /ijfw-status top-line follow this pattern |

## Tool 2: superpowers:executing-plans

Source file read:
~/.claude/plugins/cache/claude-plugins-official/superpowers/5.0.7/skills/executing-plans/SKILL.md (full)

### What it does well

**Pattern 1 -- Explicit skill announcement at entry**
First instruction: announce "I'm using the executing-plans skill to implement
this plan." This is a forced narration hook that anchors the user before any
work begins. The user always knows which skill is active.

**Pattern 2 -- TodoWrite as a contract before execution starts**
The skill requires `TodoWrite` creation before the first task executes. This
makes the task list visible and creates a shared contract: both agent and user
can see what is pending, in-progress, and complete. Mark-as-in_progress /
mark-as-completed is mandatory per task, not optional.

**Pattern 3 -- Explicit stop conditions with named trigger categories**
The skill names four stop triggers by category: blocker (missing dep, test
fails), plan gap (cannot start), unclear instruction, repeated verification
failure. Each is a named condition, not a vague "ask if confused." The user
knows exactly when the agent will pause and why.

**Pattern 4 -- Sub-skill announcement at handoff**
When execution completes, the skill announces the next sub-skill by name
before calling it: "I'm using the finishing-a-development-branch skill to
complete this work." The narration continues across the boundary.

### What is known from invocation patterns and public documentation

Superbase (also referenced as "Supabase AI assistant" in some contexts) ships
a task-list component that renders a numbered checklist with live status
indicators. Key patterns from invocation behaviour observed in peer sessions:

**Pattern 1 -- Numbered checklist with status glyphs per item**
Each task renders as: `[N] [ ] task description` or `[N] [x] task description`
for done. In-progress items: `[N] [~] task description`. The glyph set is
fixed: space = pending, x = done, ~ = in-progress.

**Pattern 2 -- Running total in the header**
Checklist header: `Tasks: 3 / 7 complete`. Updated on every state change.
The user always sees aggregate progress without reading the full list.

**Pattern 3 -- Per-item ETA or time-elapsed annotation**
Items append elapsed or estimated time in parens: `[3] [x] Write tests (1m 42s)`.
This gives the user a sense of pacing without requiring a separate timer.
